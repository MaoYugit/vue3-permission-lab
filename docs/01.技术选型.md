# `01.vue3-permission-lab` 技术选型与架构设计文档

本项目旨在通过全栈（Vue 3 + Express + MongoDB）模式，实现一套企业级、生产环境标准的**基于角色与权限点的访问控制系统（RBAC - Role-Based Access Control）**。

---

## 一、 项目架构概述

*   **项目名称**：`vue3-permission-lab`
*   **开发模式**：前后端分离，单代码仓库（Single Repo），采用独立子目录结构，无需 Monorepo 构建工具。
*   **权限模型**：**RBAC（Role-Based Access Control）**
    *   用户（User） ➡  角色（Role） ➡  权限点（Permission）。
    *   采用细粒度的权限点（如 `user:create`）控制按钮和接口，角色（如 `admin`）仅作为权限点的集合容器。

---

## 二、 前端技术选型（Frontend）

前端定位为高可用、强类型、模块化的企业级管理后台中台架构。

| 技术组件      | 选型方案              | 选型理由与专业考量                                           |
| :------------ | :-------------------- | :----------------------------------------------------------- |
| **基础框架**  | **Vue 3**             | 采用 Composition API 与 `<script setup>` 语法，提供更佳的逻辑复用性与 TypeScript 类型推导能力。 |
| **构建工具**  | **Vite 6**            | 现代前端构建标准，基于 ESM 的极速热更新（HMR），打包阶段使用 Rollup，构建效率与输出代码体积优化更佳。 |
| **编程语言**  | **TypeScript 5**      | 开启 `strict: true` 严格模式。对路由 `meta`、用户对象、API 响应数据进行全面类型声明，减少运行时潜在异常。 |
| **路由管理**  | **Vue Router 4.**     | 核心组件。利用 `router.addRoute` 动态注入权限路由；结合导航守卫（Navigation Guards）实现未登录拦截与无权访问（403）重定向。 |
| **状态管理**  | **Pinia 2**           | 相比 Vuex 更加轻量，支持原生 TypeScript 类型。采用模块化存储（如 `auth.ts`、`user.ts`），并结合持久化插件（`pinia-plugin-persistedstate`）维护 Token。 |
| **UI 组件库** | **Element Plus**      | 沉淀多年的企业级中后台组件库，开箱即用。支持图标按需引入，易于与 Vue 3 结合自定义布局和组件级权限包装。 |
| **网络请求**  | **Axios**             | 支持拦截器（Interceptors）。封装全局统一的请求拦截器（自动追加 `Authorization` Bearer Token）与响应拦截器（无感处理 `401` Token失效、`403` 权限不足及错误归一化）。 |
| **代码规范**  | **ESLint + Prettier** | 采用 ESLint Flat Config 扁平配置，配合 Prettier 进行代码风格约束，确保多端协作时的代码质量与一致性。 |

## 三、 后端技术选型（Backend）

后端定位为轻量、高效、遵循 RESTful API 规范且具备严密鉴权机制的微服务/单体服务。

| 技术组件       | 选型方案               | 选型理由与专业考量                                           |
| :------------- | :--------------------- | :----------------------------------------------------------- |
| **基础框架**   | **Express 4**          | 选用成熟度极高、中间件生态极其丰富的 Node.js 框架。通过中间件链式调用，天然适合实现接口级的权限校验拦截。 |
| **运行与开发** | **tsx + TypeScript**   | 后端同样采用 TypeScript。在开发阶段使用 `tsx` 运行代码，实现无需手动编译的极速冷启动和热重启，生产环境通过 `tsc` 编译为原生 JS 运行。 |
| **数据库**     | **MongoDB**            | 文档型 NoSQL 数据库，适合存储非结构化或半结构化的用户、角色关系。天然支持 JSON 格式，与 Node.js 配合开发效率极高。 |
| **ORM / ODM**  | **Mongoose **          | 行业标准的 MongoDB 建模工具。定义强类型的 Schema，支持数据校验（Validator）、虚拟属性（Virtuals）以及高效的 `populate` 关联查询。 |
| **身份认证**   | **jsonwebtoken (JWT)** | 无状态认证方案。登录成功后发放签名 Token，配合自定义的验证中间件实现轻量化鉴权，规避 Session 带来的分布式存储痛点。 |
| **密码安全**   | **bcrypt**             | 采用加盐哈希（Salted Hash）算法。在数据库中仅存储单向加密后的密文，防止数据库意外泄露导致的用户明文密码曝光。 |
| **接口安全**   | **cors + helmet**      | 使用 `cors` 中间件解决前后端跨域访问问题；使用 `helmet` 注入安全相关的 HTTP 头，防御常见的 web 漏洞（如 XSS、点击劫持）。 |

## 四、 数据库 Schema 深度设计

为实现专业、解耦的 RBAC，数据库设计必须支持**多对多（Many-to-Many）**的关系配置：

```text
  [User 用户]               [Role 角色]                 [Permission 权限点]
+-------------+         +-----------------+         +---------------------+
| _id         |         | _id             |         | _id                 |
| username    | *     * | name (角色名称)  | *     * | code (e.g. user:add)|
| password    |---------| code (角色标识)  |---------| name (权限名称)     |
| roles (Ref) |         | permissions(Ref)|         | description (描述)  |
+-------------+         +-----------------+         +---------------------+
```

### Mongoose Schema 定义规划：
1.  **Permission Schema**：存储系统内所有的细粒度操作。例如 `code: "user:delete"`, `name: "删除用户"`。
2.  **Role Schema**：关联 Permission 的 `_id` 数组。例如 `code: "editor"`, `permissions: [Permission1_Id, Permission2_Id]`。
3.  **User Schema**：关联 Role 的 `_id` 数组。支持单用户赋予多角色。

## 五、 核心权限流架构设计

前后端通过以下闭环逻辑协同工作，确保体验与安全的双重专业性：

```text
[前端] 登录请求 --------------------------> [后端] 校验密码，生成带有 role/id 的 JWT
[前端] 接收 Token 写入 Pinia ------------> [后端]
[前端] 请求 /api/auth/me 获取权限点 ------> [后端] 解析 Token，查询 Role 和 Permission
[前端] 1. 过滤静态路由表，动态 addRoute
       2. 根据权限过滤菜单，渲染 Sidebar
       3. v-permission 指令控制按钮显示/隐藏
[前端] 发起敏感操作 (如 DELETE /api/user) --> [后端] 校验 Token，匹配中间件 checkPermission('user:delete')
                                           [后端] 若通过则执行业务，不通过返回 403 Forbidden
```
