# 05. 登录接口（/api/auth/login）契约定义

本契约作为前后端数据交互的唯一规范，采用 TypeScript 强类型定义，确保双端在开发、打包及数据校验阶段的类型一致性。

---

## 一、 通用 API 响应包裹器（Generics Envelope）

前后端的所有数据传输必须使用符合标准格式的响应对象包裹。

### 1. 前端通用响应接口（`/frontend/src/api/types.ts`）

```typescript
/**
 * 统一 API 响应结构
 */
export interface ApiResponse<T = any> {
  code: number;      // 业务状态码（例如：200 为成功，401 为未认证，403 为无权操作等）
  message: string;   // 响应消息（用于前端展示或调试说明）
  data: T;           // 具体业务数据载荷
}
```

### 2. 后端通用响应类型（`/backend/src/types/response.ts`）

```typescript
export interface ApiResponse<T = any> {
  code: number;
  message: string;
  data: T;
}
```

---

## 二、 登录接口（`/api/auth/login`）契约详情

*   **请求路径**：`/api/auth/login`
*   **请求方法**：`POST`
*   **请求格式**：`application/json`

### 1. 请求载荷定义（Request Payload）

无论是前端发起请求，还是后端路由中间件接收，均需遵循此数据结构：

#### 前端声明（`/frontend/src/api/auth/types.ts`）及 后端声明（`/backend/src/types/auth.ts`）

```typescript
/**
 * 登录请求载荷
 */
export interface LoginRequest {
  username: string; // 长度校验逻辑需在后端及前端表单实现（建议 3-20 位字符）
  password: string; // 明文密码，由后端使用 bcryptjs 进行哈希比对
}
```

---

### 2. 响应载荷定义（Response Payload）

登录成功后，后端返回的 `data` 域中所携带的实体。

#### 前端声明（`/frontend/src/api/auth/types.ts`）及 后端声明（`/backend/src/types/auth.ts`）

```typescript
/**
 * 登录成功后的用户简要信息
 */
export interface LoginUserInfo {
  id: string;        // 用户唯一 ID（对应 MongoDB 中的 _id）
  username: string;  // 用户名
  role: string;      // 角色标识（如 'admin', 'editor', 'viewer'）
}

/**
 * 登录成功返回的实际数据结构
 */
export interface LoginResponseData {
  token: string;       // 颁发的 JWT 凭证（前端后续请求需在 Headers 的 Authorization 中携带）
  user: LoginUserInfo; // 登录用户的基本信息，用于前端状态初始化与持久化
}
```

---

### 3. 请求响应示例

#### 成功响应（HTTP 200）

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOiI2NGE4YT...",
    "user": {
      "id": "64a8a5b9c1d2e3f4a5b6c7d8",
      "username": "admin",
      "role": "admin"
    }
  }
}
```

#### 异常响应示例（HTTP 200 状态码，通过内部 code 区分业务错误）

```json
{
  "code": 400,
  "message": "用户名或密码错误",
  "data": null
}
```
