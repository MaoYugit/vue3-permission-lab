# 02. 开发顺序与垂直切片规划

本项目通过**“契约优先（Contract-First）”**与**“功能驱动的垂直切片（Feature-Driven Vertical Slicing）”**开发模式，依次攻克并验证全部 5 个维度的权限控制（页面/路由、菜单/导航栏、操作/按钮、接口/请求、视图数据/脱敏）。

---

## 一、 整体开发路线图（Roadmap）

```text
[阶段 0：环境初始化] ────> [切片 1：基础认证与接口会话] ────> [切片 2：动态路由与菜单渲染]
                                                                     │
[切片 4：数据展示与脱敏控制] <─── [切片 3：操作级权限与接口拦截] <───┘
```

---

## 二、 详细开发步骤与任务分解

### 阶段 0：项目基础环境初始化（骨架搭建）
本阶段确立项目目录结构，配置前后端独立的编译与运行环境。

*   **后端（`/backend`）初始化**：
    1.  创建 `/backend` 文件夹，执行 `pnpm init`。
    2.  安装核心依赖（`express`, `mongoose`, `dotenv`, `cors`, `helmet`, `jsonwebtoken`, `bcrypt`）。
    3.  安装开发依赖（`typescript`, `@types/node`, `@types/express`, `tsx`）。
    4.  配置 `tsconfig.json`（采用 ESM 规范）与 `package.json` 中的启动脚本。
    5.  完成 MongoDB 本地连接配置，跑通 Express 基础服务。
*   **前端（`/frontend`）初始化**：
    1.  创建 `/frontend` 文件夹，使用 `pnpm create vite . --template vue-ts` 初始化。
    2.  安装核心依赖（`vue-router@4`, `pinia`, `pinia-plugin-persistedstate`, `axios`, `element-plus`）。
    3.  安装开发依赖（`sass`, `eslint`, `prettier`）。
    4.  配置静态路由（`/login`, `/403`, `/404`）与基础页面骨架，跑通 Vite 开发服务。

---

### 切片 1：基础认证与会话控制（解决维度：④ 接口/数据请求级别权限 - 初始层）
本阶段实现用户登录流程，确保身份凭证（Token）安全流转并在请求中被正确拦截与处理。

*   **Step 1.1：定义 API 契约**
    *   定义 `/api/auth/login` 请求与响应的 TypeScript 接口规范。
*   **Step 1.2：后端实现（Authentication）**
    *   设计 Mongoose **User Schema**（包含 `username`、`password`、`role: string` 以及敏感字段 `phone: string`）。
    *   编写用户登录接口，集成 `bcrypt` 进行密码校验，验证成功后生成带有 `userId` 和 `role` 载荷的 JWT。
*   **Step 1.3：前端实现（Session & Interceptors）**
    *   编写登录页面（Login.vue）并配置 Pinia Store（`useAuthStore`）保存 Token 状态（持久化至 `localStorage`）。
    *   **配置 Axios 请求拦截器**：自动在 Header 中注入 `Authorization: Bearer <token>`。
    *   **配置 Axios 响应拦截器**：捕获 `401` 状态码（未认证/凭证过期），清除 Token 并强制重定向至 `/login`。
*   **Step 1.4：端到端（E2E）联调验证**
    *   前端登录成功获取 Token，观察后续任意请求的 Request Headers 中是否成功携带 `Bearer <token>`。

---

### 切片 2：动态路由与菜单渲染（解决维度：① 页面/路由级别权限、② 菜单/导航栏级别权限）
本阶段实现基于角色的动态菜单生成与页面防跨权访问。

*   **Step 2.1：定义完整 RBAC 契约与初始化数据**
    *   定义 `/api/auth/me`（获取当前用户信息与权限集）的契约。
    *   设计 Mongoose **Role Schema** 与 **Permission Schema**。
    *   编写后端 Seed 脚本初始化预置数据：
        *   **Permissions**：`user:read`（读权限）、`user:write`（增改权限）、`user:delete`（删除权限）、`user:sensitive`（敏感数据查看权限）。
        *   **Roles**：`admin` (含全部权限), `editor` (含 read/write), `viewer` (仅含 read)。
        *   **Users**：创建不同角色对应的测试账号。
*   **Step 2.2：后端实现（Profile API）**
    *   编写 `/api/auth/me` 接口。解析 JWT 并通过 `populate` 查询用户关联的 Role 和 Permissions，向前端返回当前用户的 `roles` 和 `permissions`（权限码数组）。
*   **Step 2.3：前端实现（Dynamic Routes & Menu Guard）**
    *   划分**静态路由表**与**动态路由表**。在动态路由的 `meta.permissions` 中定义所需权限码。
    *   编写 Vue Router 全局前置守卫（`beforeEach`）：
        *   登录拦截与用户信息补全。
        *   根据获取到的 `permissions` 权限码，过滤动态路由表，通过 `router.addRoute` 动态注入。
    *   根据过滤后最终生效的路由表，结合 Element Plus 菜单组件，动态渲染左侧导航栏。
*   **Step 2.4：端到端（E2E）联调验证**
    *   使用不同账号登录，验证菜单展示差异；在地址栏手动输入无权访问的 URL，验证是否被重定向至 `/403`。

---

### 切片 3：操作级控制与接口拦截（解决维度：③ 操作/按钮级别权限、④ 接口/数据请求级别权限 - 保护层）
本阶段实现页面内具体按钮的安全隐藏和后端对应的 API 强防护。

*   **Step 3.1：前端按钮控制（v-permission）**
    *   编写自定义指令 `v-permission`。对比当前用户的权限集，若无对应权限点，在 `mounted` 时通过 DOM 操作将其从父节点中移除。
    *   封装全局函数 `hasPermission` 以支持复杂的逻辑判断条件（如某些按钮的 `disabled` 属性状态控制）。
*   **Step 3.2：后端接口拦截（Middleware）**
    *   编写权限校验中间件 `checkPermission(requiredPermission: string)`。
    *   在敏感业务接口（如 `DELETE /api/users/:id`）中挂载中间件，校验请求用户的权限集，不通过则直接返回 `403 Forbidden`。
*   **Step 3.3：端到端（E2E）安全验证**
    *   使用 `viewer` 账号登录，确认对应删除按钮已不可见。手动通过 Postman 直接调用删除接口，验证后端是否正确返回 403 拦截。

---

### 切片 4：数据级展示与敏感数据脱敏（解决维度：⑤ 视图数据展示/脱敏级别权限）
本阶段实现根据权限点对返回的视图数据进行细粒度脱敏和界面展示差异化。

*   **Step 4.1：定义敏感数据查询契约**
    *   定义 `/api/users`（获取用户列表，含手机号等敏感信息）的契约。
*   **Step 4.2：后端实现（Data Masking Middleware）**
    *   编写用户列表查询接口，并在对应的路由处理器中加入数据脱敏逻辑：
        *   获取当前请求用户的权限集。
        *   若用户**不拥有** `user:sensitive` 权限点，则在将数据返回前，对 `phone` 字段执行脱敏处理（例如：将 `13812345678` 转换为 `138****5678`）。
        *   若用户**拥有**该权限点，则原样返回完整数据。
*   **Step 4.3：前端实现（View Control）**
    *   前端 UserList 页面渲染用户表格，包含手机号列。
    *   利用 `hasPermission('user:sensitive')` 函数控制表格列或操作提示：若无敏感信息查看权限，在手机号列旁展示“无权查看完整信息”的提示 UI；若有，则展示“复制”按钮等功能。
*   **Step 4.4：端到端（E2E）联调验证**
    *   分别使用 `admin`（拥有 `user:sensitive`）与 `editor`（不拥有）账号访问用户列表页面。
    *   观察控制台 Network 面板，确认接口返回的原始 JSON 数据本身已经过脱敏处理，同时验证前端界面的差异化呈现。
